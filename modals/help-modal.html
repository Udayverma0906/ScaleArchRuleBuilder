Guide

How ScaleArch Rule Builder works

1
Fill in the rule identity
Give your rule a category, a kebab-case name (e.g. no-raw-sql), a short message (shown in the squiggly tooltip in VS Code), and a detailed hint explaining why the code is bad and how to fix it. The hint is what developers see on hover — make it actionable.
2
Choose Regex or AST
Regex — detects text patterns on a single line. Fast. Good for 80% of rules.
AST — understands code structure. Use only when you need to know about function length, class methods, nesting depth, or parameter counts. Not sure? See sections 02 and 03 below.
3
Pick from the library or write your own
The pattern library has 54 ready-made patterns. Search by name or filter by category. Click Use → to pre-fill the entire form. If nothing fits, write your own pattern in the Regex pattern field and use Test → to verify it before generating.
4
Pick a target language
Use All languages for rules that apply everywhere (e.g. SQL patterns, hardcoded secrets). Use a specific language for patterns that only make sense in that language — e.g. System.out.println is Java-only, bare except: is Python-only.
5
Generate, verify, then paste
Hit Generate rule code. Click Verify with Claude → to get a second opinion before using it. When you're confident, Copy and paste the code into src/rules/customRules.ts in your ScaleArch extension. Press F5 in VS Code to test it live.
✓ Use Regex when…
  • The problem is visible on one line of code
  • You're looking for a specific function call (eval(), console.log())
  • You're detecting SQL patterns in strings
  • You're banning a specific import or API
  • You're detecting hardcoded values (secrets, IPs)
✗ Don't use Regex when…
  • You need to know how long a function is
  • You need to count methods in a class
  • You need to check nesting depth
  • The pattern spans multiple lines
  • You need to understand code structure
Built-in example — py/bare-except
Catches Python's bare except: which silently swallows every exception including KeyboardInterrupt. A single line is enough to detect it.
{
  id: 'py/bare-except',
  category: 'code-quality',
  languages: ['python'],
  severity: vscode.DiagnosticSeverity.Warning,
  message: 'Bare except: catches everything including KeyboardInterrupt',
  hint: 'Use except ValueError: or except (TypeError, ValueError):',
  test: (line) => /^\s*except\s*:/.test(line),
}
Your own example — ban a deprecated internal API
Your company deprecated PaymentServiceV1. Anyone who writes it gets an instant red squiggly with a link to the migration guide.
{
  id: 'company/no-payment-v1',
  category: 'security',
  languages: ['typescript', 'javascript'],

}

AST stands for Abstract Syntax Tree. When ScaleArch parses your file, it builds a tree of every function, class, loop, and expression. An AST rule receives one node at a time from that tree and decides whether to flag it.

Think of it like this: Regex reads your code as text. AST reads it as code.

What a simple function looks like as an AST
function greet(name) {   →   FunctionDeclaration
  return "Hello " + name;       id.name = "greet"
}                                params = [name]
                                 body.body = [ReturnStatement]
✓ Use AST when…
  • You need to know function length in lines
  • You need to count methods in a class
  • You need to check parameter count
  • You need to detect empty catch blocks
  • You need to check nesting depth
✗ Don't use AST when…
  • A regex on the line would work fine
  • You're looking for a string pattern
  • You want to detect a function call by name
  • You want something fast and simple
Built-in example — solid/srp (JS/TS AST)
Flags any class with more than 10 methods — a sign it's doing too much (Single Responsibility Principle). Regex cannot count methods; AST can.
export function checkSRP(node: any): RuleResult | null {
  if (node.type !== 'ClassDeclaration') return null;

  const methods = node.body.body.filter(
    m => m.type === 'MethodDefinition' && m.kind !== 'constructor'
  );
  if (methods.length <= 10) return null;

  return {
    range: makeRange(node),
    code: 'solid/srp',
    message: `Class has ${methods.length} methods — consider splitting`,
    hint: 'A class should have one reason to change.',
    severity: vscode.DiagnosticSeverity.Warning,
  };
}
Your own example — every async function must have try/catch
Company policy: unhandled promise rejections crash the server. This rule flags any async function without a try/catch block.
function checkAsyncMustHaveTryCatch(node: any): RuleResult | null {
  const fnTypes = ['FunctionDeclaration','FunctionExpression','ArrowFunctionExpression'];
  if (!fnTypes.includes(node.type)) return null;
  if (!node.async) return null;

  const hasTryCatch = (node.body?.body ?? []).some(
    s => s.type === 'TryStatement'
  );
  if (hasTryCatch) return null;

  return {
    range: makeRange(node),
    code: 'company/async-no-try-catch',
    message: 'Async function has no try/catch — unhandled rejections crash the server',
    hint: 'Wrap the body in try/catch or use your withErrorHandling() wrapper.',
    severity: vscode.DiagnosticSeverity.Error,
  };
}

Tip: Paste any TypeScript or JavaScript into astexplorer.net and select @typescript-eslint/parser — this is the exact parser ScaleArch uses. Click any node to see its type and all available properties.

Section 4 was added to customRules.ts in v1.2.0 to support custom Python AST rules. It works identically to Section 2 (JS/TS AST) but uses the PythonRuleCheck type and the CUSTOM_PYTHON_AST_RULES array.

Generated code from this builder (when Python AST language is selected) is ready to paste directly into Section 4.

✓ Python node properties
  • node._type — node type (underscore prefix)
  • node.lineno / node.end_lineno — line numbers (1-based)
  • node.name — function or class name string
  • node.args.args — list of parameter nodes
  • node.body — flat list of child statements
✗ JS/TS patterns that break in Python
  • node.type → use node._type
  • node.loc.start.line → use node.lineno
  • node.id?.name → use node.name
  • node.params → use node.args.args
  • node.body.body → use node.body
Python AST example — detect empty except blocks
Flags Python except blocks that only contain pass — silently swallowing the exception.
function checkPyEmptyExcept(node, _cfg, makeDiag) {
  if (node._type !== 'ExceptHandler') return null;

  const body = node.body ?? [];
  const isOnlyPass = body.length === 1 && body[0]._type === 'Pass';
  if (!isOnlyPass) return null;

  return makeDiag(
    node,
    'Empty except block — exception silently swallowed',
    vscode.DiagnosticSeverity.Warning,
    'custom/py-empty-except',
    'Bare pass in except hides bugs. At minimum log the error: except Exception as e: logger.error(e)'
  );
}

// Register in customRules.ts Section 4:
export const CUSTOM_PYTHON_AST_RULES: PythonRuleCheck[] = [
  checkPyEmptyExcept,   // ← add here
];

Section 5 was added to customRules.ts in v2.0.0 to support custom Java AST rules. Java uses tree-sitter-java — a native Node.js binding that parses Java without requiring a JDK.

Java rules are declared as named consts assigned to the JavaRuleCheck type, then the const name is added to the CUSTOM_JAVA_AST_RULES array — same pattern as Python and JS/TS.

Generated code from this builder (when Java AST language is selected) is ready to paste directly into Section 5.

✓ tree-sitter node properties
  • node.type — node type, NO underscore (snake_case)
  • node.startPosition.row — start line (0-based)
  • node.endPosition.row — end line (0-based)
  • node.childForFieldName('name')?.text — get named field
  • node.namedChildren — children excluding syntax tokens
  • node.parent — parent node (for context checks)
✗ Python / JS/TS patterns that break in Java
  • node._type → use node.type (no underscore)
  • node.name → use node.childForFieldName('name')?.text
  • node.id?.name → use node.childForFieldName('name')?.text
  • node.body.body → use node.namedChildren
  • node.loc.start.line → use node.startPosition.row
  • node.params → use node.childForFieldName('parameters')?.namedChildren
Common Java node types (tree-sitter snake_case)
method_declaration      → public void myMethod(...) { }
class_declaration       → public class MyClass { }
constructor_declaration → public MyClass(...) { }
catch_clause            → } catch (Exception e) { }
field_declaration       → private String name;
interface_declaration   → public interface MyService { }
method_invocation       → System.exit(0);
binary_expression       → status == "active"
enhanced_for_statement  → for (String item : items) { }
if_statement            → if (condition) { }
Example 1 — empty catch block
Flags catch blocks with no statements — silently swallowing exceptions is a common Java anti-pattern.
const checkEmptyCatch: JavaRuleCheck = (node, _cfg, makeDiag) => {
  if (node.type !== 'catch_clause') return null;

  const body = node.childForFieldName('body');
  if (!body) return null;
  if (body.namedChildren.length > 0) return null;  // not empty

  return makeDiag(
    node,
    'Empty catch block — exception is being silently swallowed',
    vscode.DiagnosticSeverity.Error,
    'custom/java-empty-catch',
    'Silent catch blocks hide bugs. At minimum: catch (Exception e) { logger.error("...", e); }'
  );
};
Example 2 — too many parameters
Flags methods with more than 4 parameters — a sign that a parameter object or Builder pattern is needed.
const checkTooManyParams: JavaRuleCheck = (node, cfg, makeDiag) => {
  if (node.type !== 'method_declaration') return null;

  const params = node.childForFieldName('parameters');
  if (!params) return null;
  const count = params.namedChildren.filter(
    (c: JavaNode) => c.type === 'formal_parameter'
  ).length;
  if (count <= 4) return null;

  const name = node.childForFieldName('name')?.text ?? 'unknown';
  return makeDiag(
    node,
    `Method '${name}' has ${count} parameters — use a parameter object`,
    vscode.DiagnosticSeverity.Warning,
    'custom/java-too-many-params',
    'Methods with 5+ parameters are hard to call correctly and easy to misorder. Use a builder or parameter object.'
  );
};

No JDK required. tree-sitter-java is a native Node.js module — it parses Java source code directly without needing Java installed.

Line numbers are 0-based. node.startPosition.row is 0-based — VS Code Range is also 0-based, so no adjustment is needed. This is different from Python where lineno is 1-based.

makeDiag signature: makeDiag(node, message, severity, code, hint) — hint is always the 5th argument, never put it in a return object.
